# 🫙 Баночки

**Семейный цифровой погреб для домашних заготовок**

Что заготовили, где лежит и сколько осталось — без таблиц, бумажек и загадочного «кажется, на даче ещё была смородина».

`Flutter` · `iOS 13+` · `Android 7+` · `local-first` · `без аккаунта и интернета`

</div>

> **Статус проекта:** реализованы **R1 — Local foundation** и **R2 — QR workflow**. Приложение локально ведёт партии, места, остатки, историю и QR-этикетки. Семейная синхронизация, голос и рецепты остаются следующими этапами.

## Зачем нужны «Баночки»

Одна запись в приложении соответствует **партии** заготовок: например, 18 банок клубничного варенья, сделанных 12 июля и поставленных на вторую полку погреба. Когда банку взяли, вернули или списали как испорченную, приложение добавляет событие и сразу пересчитывает остаток.

Данные хранятся на устройстве. Для основного сценария не нужны сервер, регистрация или стабильный интернет — даже если погреб уверенно экранирует цивилизацию.

## Возможности R1/R2

| Возможность | Что получает пользователь |
| --- | --- |
| Семья и участник | Быстрый старт без аккаунта и номера телефона |
| Дерево мест | Дом → Кладовая → Стеллаж → Полка, до 6 уровней |
| Партии заготовок | Название, категория, дата, количество, единица, место и заметка |
| Живой остаток | Действия «Взял», «Вернул», «Испорчено» и уточнение факта |
| История | Неизменяемый журнал событий по каждой партии |
| Каталог | Поиск, фильтры, сортировка и просмотр архивных партий |
| Перемещение | Смена места без потери истории |
| Крупный режим | Большие цели касания, упрощённая навигация и тот же набор данных |
| Темы | Светлая, тёмная или системная |
| Надёжность | SQLite, транзакции, idempotency и полная пересборка проекций |
| QR партии и места | Версионируемый opaque QR без данных семьи, партии или остатка |
| Короткий номер | Формат `XXXXXX-C` с Luhn checksum и локальным поиском без камеры |
| Свободные этикетки | Генерация листа и явная подтверждаемая привязка к партии или месту |
| Печать | PDF preview/share/print: одна, средняя и мелкая A4-сетка; кириллица встроена локально |

## Как это выглядит в жизни

1. Создайте семью и укажите имя первого участника.
2. Добавьте место `Дача → Погреб → Полка 2`.
3. Создайте партию: `Огурцы маринованные`, 18 банок.
4. Нажмите «Взял 2», затем «Вернул 1» и «Испорчено 1».
5. На карточке останется **16 банок**, а в истории будут видны все четыре операции.

Если фактическое количество разошлось с расчётным, используйте «Уточнить остаток». Исходные события не переписываются: приложение добавит корректирующее событие и сохранит след действий.

## Быстрый старт

### 1. Подготовьте окружение

Нужен Flutter `3.44.0` / Dart `3.12.0` или совместимая более новая stable-версия.

```bash
flutter doctor -v
```

Для iOS также нужны macOS и Xcode, для Android — Android Studio или Android SDK с настроенным эмулятором/устройством.

### 2. Установите зависимости

```bash
cd apps/mobile
flutter pub get
```

### 3. Выберите устройство и запустите приложение

```bash
flutter devices
flutter run -d <device-id>
```

Если подключено только одно устройство, достаточно `flutter run`.

Подробные команды для iOS Simulator, Android Emulator, debug-сборок и решения частых проблем собраны в [гайде по запуску](docs/GETTING_STARTED.md).

## Первый запуск

Есть два варианта:

- **чистый сценарий** — введите название семьи и имя участника, затем создайте места и первую партию;
- **готовый пример** — в debug-сборке нажмите **«Открыть debug-пример»**. Приложение создаст семью, дерево мест, партию и несколько событий. Повторно добавить пример можно в настройках.

Debug-данные никогда не создаются автоматически и недоступны в release-сборке.

## Как устроен проект

```text
apps/mobile/
├── lib/app/                 # bootstrap и Riverpod application state
├── lib/core/                # SQLite, ошибки, UUID, часы, UI-токены
├── lib/features/            # feature-first presentation/domain/data
├── test/                    # unit, database и widget tests
└── integration_test/        # сквозной persistent local flow

docs/
├── GETTING_STARTED.md       # подробный запуск и troubleshooting
├── ARCHITECTURE.md          # границы слоёв и поток данных
├── DATA_MODEL.md            # таблицы, события, projections
├── OFFLINE_FIRST.md         # гарантии local-first
├── ACCESSIBILITY.md         # крупный режим и доступность
├── TESTING.md               # стратегия и матрица проверок
└── adr/                     # зафиксированные архитектурные решения
```

Основной поток данных:

```text
Flutter UI → Riverpod controller → Repository → SQLite
                                      ↓
                              события → projections
```

Сетевых SDK и runtime-зависимости от backend в R1/R2 нет.

## Важные ограничения R1/R2

- Удаление приложения удаляет локальную базу; облачного восстановления пока нет.
- SQLite-файл находится в app-private storage и защищён средствами ОС, но отдельный SQLCipher-слой ещё не добавлен.
- Несколько устройств не синхронизируются — это задача R3.
- QR работает полностью локально; без R3 код с другого устройства может быть валиден, но отсутствовать в локальной SQLite базе.
- Голос и рецепты сознательно не спрятаны в интерфейсе «на будущее»: сейчас они не реализованы.

Архитектурные причины и границы этих решений зафиксированы в [`docs/adr`](docs/adr).
